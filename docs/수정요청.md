# 수정 요청 가이드 (AI 에이전트에게 수정을 맡기는 방법)

이 문서는 [`docs/ANALYSIS.md`](./ANALYSIS.md)에서 발견된 항목들을 **다른 AI(Claude Code, Copilot, Cursor 등)에게 수정 요청할 때 그대로 복사해서 쓸 수 있는 프롬프트**와, 요청 시 반드시 함께 전달해야 하는 컨텍스트를 정리한 것입니다.

---

## 0. 모든 수정 요청에 공통으로 포함할 컨텍스트

AI에게 수정을 요청할 때, 아래 블록을 프롬프트 맨 앞에 붙여 주세요. 이 규칙을 빠뜨리면 AI가 플러그인의 스레드 모델이나 독립성 원칙을 깨는 수정을 할 수 있습니다.

```text
[프로젝트 규칙 — 반드시 지킬 것]
- 이 저장소는 Unreal Engine 디지털 트윈용 공통 런타임 플러그인(DTCore)이다.
- ParseToStruct()는 백그라운드 스레드에서 실행된다. 내부에서 NewObject, GetWorld(),
  Actor/Component/Blueprint 접근을 절대 하면 안 된다. 순수 파싱만 허용된다.
- ProcessStructData()만 GameThread에서 실행되며, UObject 접근은 여기서만 한다.
- 특정 프로젝트 모듈/맵/Actor/서버 URL에 의존하는 코드를 넣지 않는다.
  환경별 값은 UDTCoreSettings 또는 Game.ini [DTCoreRuntimeOverride]로만 주입한다.
- Public 헤더에 새 의존성을 추가하지 않는다. (Public 모듈: Core, CoreUObject, Engine, UMG만)
- Source/DTCore/Private/Lib/yyjson.c, Public/Lib/yyjson.h 는 서드파티 코드이므로 수정 금지.
- Base 클래스들은 Tick이 기본 비활성화(bStartWithTickEnabled=false) 상태를 유지해야 한다.
- 자세한 아키텍처는 README.md와 docs/ANALYSIS.md 참고.
```

## 1. [High] Api 토픽 라우팅 구조 정리 (ANALYSIS 4.1)

- 대상: `Source/DTCore/Private/Core/DxWebSocketSubsystem.cpp:178-186`
- 문제: 커밋 `2eec1fe`가 `ETopicRouteType::Api` 분기를 `EnqueueWebSocketData`로 변경하여, 두 분기가 동일해지고 `WebSocketTopics`/`ApiTopics` 설정 구분이 무의미해짐.
- **먼저 결정할 것 (사람이 결정)**: 이 변경이 (A) 임시 조치였고 원래대로 `EnqueueApiData`로 복구해야 하는지, (B) 확정 구조라서 라우팅 분기 자체를 제거해야 하는지.

### (A) 원복이 맞는 경우의 요청 프롬프트

```text
DxWebSocketSubsystem.cpp의 ReceivedMessage()에서 ETopicRouteType::Api 분기가
EnqueueWebSocketData를 호출하고 있는데, 이건 커밋 2eec1fe에서 잘못 바뀐 것이다.
EnqueueApiData(BodyString) 호출로 복구해줘.
복구 후 README.md 4.2절의 Topic 라우팅 표가 코드와 일치하는지 확인하고,
다르면 "ApiTopics → EnqueueApiData"로 표도 함께 맞춰줘.
다른 코드는 수정하지 마.
```

### (B) 현재 동작이 확정인 경우의 요청 프롬프트

```text
DxWebSocketSubsystem에서 모든 토픽이 EnqueueWebSocketData로 가는 것이 확정 구조다.
이에 맞춰 다음을 정리해줘:
1. ETopicRouteType enum과 TopicRouteMap의 Api/WebSocket 구분이 여전히 필요한지 검토하고,
   불필요하면 단순화(등록된 토픽인지 여부만 확인)해줘.
2. UDTCoreSettings의 ApiTopics 설정이 여전히 의미가 있는지 검토해줘.
   (구독 대상 목록으로는 여전히 쓰이므로 제거하지 말고, 주석으로 용도를 명확히 해줘)
3. README.md 4.2절 라우팅 설명을 실제 동작과 일치하게 수정해줘.
호환성을 위해 UDTCoreSettings의 기존 프로퍼티 이름은 바꾸지 마.
```

## 2. [Medium] ParseToStruct 스레드 규칙 코드 가드 (ANALYSIS 4.4)

- 대상: `Source/DTCore/Public/Api/ApiMessage.h`, `Source/DTCore/Public/WebSocket/TransactionCodeMessage.h` 및 호출부 `DxDataSubsystem.cpp`

### 요청 프롬프트

```text
UApiMessage::ParseToStruct와 UTransactionCodeMessage::ParseToStruct는 백그라운드
스레드에서 호출되는 계약인데, 파생 클래스가 이를 어겨도 감지할 방법이 없다.
다음을 추가해줘:
1. 두 함수의 헤더 선언부에 "백그라운드 스레드에서 호출됨, UObject 생성/GameThread
   전용 API 사용 금지"를 명시하는 주석(Doxygen 스타일).
2. DxDataSubsystem에서 ParseToStruct를 호출하기 직전에 개발 빌드 한정으로
   ensure 또는 checkf(!IsInGameThread(), ...) 수준의 가드가 아니라,
   호출 컨텍스트를 검증하는 것이므로 반대로 백그라운드 스레드임을 확인하는
   디버그 체크를 넣어줘 (Shipping 빌드에는 비용이 없어야 함).
런타임 동작(파싱 결과)은 절대 바뀌면 안 된다.
```

## 3. [Medium] ApiDataTable 로드 책임 일원화 (ANALYSIS 4.2)

- 대상: `Source/DTCore/Private/Core/DxApiSubsystem.cpp:17-21`, `Source/DTCore/Private/Core/DxDataSubsystem.cpp:22-34`

### 요청 프롬프트

```text
UDxApiSubsystem과 UDxDataSubsystem이 각각 UDTCoreSettings->ApiDataTable을
LoadSynchronous()로 따로 로드하고 있다. 로드 책임을 한 곳으로 일원화해줘.
- Subsystem 초기화 순서 의존성이 생기지 않게 주의할 것
  (GameInstanceSubsystem 간 Initialize 순서는 보장되지 않으므로,
   Collection.InitializeDependency<T>()를 사용해 명시적으로 의존성을 선언할 것).
- 실패 시 기존과 동일하게 DX_LOG 경고를 남길 것.
- 외부(소비 프로젝트)에서 보이는 API 시그니처는 바꾸지 말 것.
```

## 4. [Medium] 핵심 순수 로직 Automation Test 도입 (ANALYSIS 4.6)

### 요청 프롬프트

```text
DTCore 플러그인에 Unreal Automation Test(IMPLEMENT_SIMPLE_AUTOMATION_TEST 기반)를
도입해줘. 우선 순수 로직만 대상으로 한다:
1. FYyJsonParser: 정상 JSON, 깨진 JSON, meta.resource/action 추출, MESSAGE_ID 추출.
2. DTCoreRuntimeConfig: [DTCoreRuntimeOverride] 우선순위와 빈 값 무시 규칙.
3. 핸들러 키 생성 규칙 (FPaths::Combine(Resource, Action)).
테스트 코드는 Source/DTCore/Private/Tests/ 아래에 두고, WITH_DEV_AUTOMATION_TESTS
가드로 감싸서 Shipping 빌드에 포함되지 않게 해줘.
기존 런타임 코드는 테스트를 위해 최소한으로만 변경(예: private 함수 노출이 필요하면
friend 대신 protected + 테스트 전용 서브클래스 사용)해줘.
```

## 5. [Low] PascalCase 죽은 코드 정리 (ANALYSIS 4.3)

- 대상: `Source/DTCore/Private/Core/DxDataSubsystem.cpp:48-51`

### 요청 프롬프트

```text
DxDataSubsystem.cpp의 Initialize()에서 "PascalCase로 변환"이라는 주석 아래
실제 변환 없이 변수만 복사하고 있다 (PascalResource/PascalAction).
대소문자 변환은 하지 않는 것이 현재 확정 동작이므로:
1. 오해를 부르는 주석과 불필요한 중간 변수를 제거하고
   Key = FPaths::Combine(Row.ApiResource, Row.ApiAction) 으로 단순화해줘.
2. 대신 "DataTable의 Resource/Action 표기는 서버 응답 meta.resource/action과
   대소문자까지 일치해야 한다"는 주석을 남겨줘.
키 생성 결과가 기존과 완전히 동일해야 한다 (동작 변경 금지).
```

DTCoreSettings.h의 WebSocketLogin/WebSocketPasscode 기본값이 "artemis"로

## 7. 수정 완료 후 공통 검증 절차

AI가 수정을 마치면 아래를 확인하세요 (README 11절 체크리스트 기반):

```text
1. 소비 프로젝트에 플러그인을 넣고 정상 빌드되는지 (플러그인 단독 빌드는 불가)
2. ApiDataTable / WebSocketDataTable이 정상 로드되는지 (로그 확인)
3. WebSocket 연결/재연결 로그가 정상 출력되는지
4. WebSocket 메시지가 Topic 설정에 따라 올바른 큐로 들어가는지
5. ParseToStruct가 백그라운드 스레드에서 크래시 없이 동작하는지
6. Game.ini override 값 채움/비움 시 접속 정보 fallback이 기대대로인지
7. git diff로 요청 범위 밖의 파일이 변경되지 않았는지
```

## 8. 요청 시 주의사항

- **한 번에 하나의 항목만** 요청하는 것을 권장합니다. 여러 항목을 섞으면 diff 검토가 어려워집니다.
- 항목 1(라우팅)은 (A)/(B) 결정이 선행되어야 합니다. 결정 없이 AI에게 맡기지 마세요.
- 수정 후 커밋 메시지에 대상 항목 번호(예: "ANALYSIS 4.3 정리")를 남기면 추적이 쉽습니다.
